## 一键启动后端栈（dify 风格）

用户希望和 dify 一样：`cd docker && cp .env.example .env && docker compose up -d --build` 启动后端 + 所有依赖。

### 已落盘
- `server/Dockerfile` — 多阶段 (base/deps/dev/builder/prod)，alpine + openssl + libpq
- `server/entrypoint.sh` — 等 DB → prisma generate → migrate deploy 或 db push → node dist/index.js
- `docker/docker-compose.yml` — 编排 postgres/redis/minio/minio-init/server(prod)/server-dev；profile 拆分 prod 和 dev
- `docker/.env.example` — 所有可调参数集中
- `docker/minio-init.sh` — 用 mc 自动建 bucket
- `docker/README.md` — 一键启动 + dev 模式 + 排错文档
- `.gitignore`（项目根）— 忽略 `dockerCache/` 和 `docker/.env`

### 关键决策
- **dev/prod 拆 profile**：prod 走编译后 node 跑，dev 用 ts-node-dev + bind mount 源码做热重载
- **迁移策略**：entrypoint 检查 `_prisma_migrations` 表是否存在，存在则 `migrate deploy`，否则 `db push`（首次空库）
- **依赖顺序**：`depends_on.condition: service_healthy` + minio-init 走 `service_completed_successfully`，db 真正就绪才起 server
- **服务间通信**：compose 内部用 service 名（`postgres`/`redis`/`minio`），宿主机用 `localhost` + 端口
- **数据卷**：`../dockerCache/<service>Cache`（项目根的 dockerCache/），按用户要求
- **Redis 密码**：以 `.env.example` 为准（空），与代码一致

### 端口
3000 / 5432 / 6379 / 9000 / 9001（API/Console）

### 启动命令
```bash
cp docker/.env.example docker/.env
docker compose --profile prod up -d --build
```

## docker compose one-shot job 的状态判断
`minio-init` 这类 `restart: "no"` 的容器，跑完后状态是 `Exited (0)`，**这是正确的，不是 bug**。判断标志：
- `Exited (0)` = 任务成功（看 server 是否正常起）
- `Exited (非 0)` = 失败（查 logs）
- `Restarting` = 真有问题

日志里出现两次（如 "Creating bucket" + "already exists"）通常是用户重跑了 `docker compose up -d` 或 server 第一次没起来重试，因为脚本有 `|| true` 兜底所以幂等。

## Prisma 7 兼容性（踩坑）
**`prisma db push` 在 7.x 不再支持 `--skip-generate` 参数**——传了就 `unknown or unexpected option: --skip-generate` 退出（exit code 1），导致 server 容器不停重启。

Prisma 7 的 db push 选项：`-h` / `--config` / `--schema` / `--url` / `--accept-data-loss` / `--force-reset`。
不要再加 `--skip-generate`，反正上面 entrypoint 已经先 `prisma generate` 过了。

同类潜在风险：`migrate dev` 在 7.x 也精简了命令集，以后升级遇到 unknown option 优先查 prisma changelog。

## DOCKER_BUILDKIT=0 绕开 buildx 权限问题

macOS Docker Desktop 的 buildx 经常报 `failed to update builder last activity time: open /Users/qsj/.docker/buildx/activity/.tmp-xxx: operation not permitted`。用 `DOCKER_BUILDKIT=0` 环境变量切换经典 builder 解决：
```bash
DOCKER_BUILDKIT=0 docker compose --profile prod build server
```

## Prisma 7 需要 prisma.config.ts 在项目根

Prisma 7 不再在 schema.prisma 的 datasource 块写 url，而是拆到 `prisma.config.ts`（项目根，和 package.json 同级）：
```ts
import { defineConfig } from "prisma/config";
export default defineConfig({
  datasource: { url: process.env.DATABASE_URL },
});
```
注意放在 `prisma/` 目录下不行——Prisma CLI 只找根目录的 `prisma.config.ts`。

## 修复 App Sandbox 阻止 localhost 网络请求 (2026-06-24)

**症状**: 运行时大量 `Sandbox does not allow access` / `HTTP load failed -1003` / 二维码无法加载

**根因**: Xcode 项目默认 `ENABLE_APP_SANDBOX = YES`，macOS 沙箱拦截了所有网络 + DNS 请求

**修复**:
1. 创建 `wallpapersetter-ui.entitlements` — 显式 `com.apple.security.app-sandbox = false` + `network.client = true`
2. 创建 `Info.plist` — `NSAllowsLocalNetworking = YES`
3. `project.pbxproj` — `ENABLE_APP_SANDBOX → NO`，加 `CODE_SIGN_ENTITLEMENTS`，加 `INFOPLIST_KEY_NSAppTransportSecurity`

**结果**: 现在可以直接调用 `localhost:3000` 的后端 API，无需额外注册微信服务。linkd/SetStoreUpdate/dnssd 等系统级错误可忽略。
