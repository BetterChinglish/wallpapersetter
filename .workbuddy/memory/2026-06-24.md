## 一键启动后端栈（dify 风格）

用户希望和 dify 一样：`cd docker && cp .env.example .env && docker compose up -d --build` 启动后端 + 所有依赖。

### 已落盘
- `server/Dockerfile` — 多阶段 (base/deps/dev/builder/prod)，alpine + openssl + libpq
- `server/entrypoint.sh` — 等 DB → prisma generate → migrate deploy 或 db push → node dist/index.js
- `docker/docker-compose.yml` — 编排 postgres/redis/minio/minio-init/server(prod)/server-dev；profile 拆分 prod 和 dev
- `docker/.env.example` — 所有可调参数集中
- `docker/minio-init.sh` — 用 mc 自动建 bucket
- `docker/README.md` — 一键启动 + dev 模式 + 排错文档
- `.gitignore`（项目根）— 忽略 `dockerCache/` 和 `docker/.env`

### 关键决策
- **dev/prod 拆 profile**：prod 走编译后 node 跑，dev 用 ts-node-dev + bind mount 源码做热重载
- **迁移策略**：entrypoint 检查 `_prisma_migrations` 表是否存在，存在则 `migrate deploy`，否则 `db push`（首次空库）
- **依赖顺序**：`depends_on.condition: service_healthy` + minio-init 走 `service_completed_successfully`，db 真正就绪才起 server
- **服务间通信**：compose 内部用 service 名（`postgres`/`redis`/`minio`），宿主机用 `localhost` + 端口
- **数据卷**：`../dockerCache/<service>Cache`（项目根的 dockerCache/），按用户要求
- **Redis 密码**：以 `.env.example` 为准（空），与代码一致

### 端口
3000 / 5432 / 6379 / 9000 / 9001（API/Console）

### 启动命令
```bash
cp docker/.env.example docker/.env
docker compose --profile prod up -d --build
```

## 首次构建踩坑（2026-06-24）

### Dockerfile 修复
**问题 1：** `prisma generate` 在 `npm ci` 之前跑，报 "Could not resolve @prisma/client"
**修法：** 调换顺序，`npm ci` 在前、`prisma generate` 在后

### 源码 TS 错误修复
1. `package.json` 缺 `@types/jsonwebtoken` 和 `@types/multer`（不写进 manifest，docker `npm ci` 会清掉）
2. `utils/minio.ts` 加 `export const BUCKET_NAME`
3. `routes/auth.ts` `redis.setex` → `setEx`（redis@6 重命名）
4. `routes/wallpapers.ts`:
   - `prisma.wallpaper.create` data 字段对齐 schema（`fileUrl`→`contentUrl`，删 `fileKey`/`tags`/`isPublic`，`userId`→`authorId`，加 `type:'WEB'`/`isApproved:false`）
   - `req.params.id` 用 `String()` 强转避免 `string|string[]` 报错
   - `getFileUrl` 加 await + bucketName 参数

### 关键教训
**修 TS 错误时一定要把 devDependencies 改进 package.json，不能只 `npm i` 一下。** docker 构建走 `npm ci`，看到 package.json 才是真相。
